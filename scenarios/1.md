* クライアント →（HTTPS）→ EC2上のPythonアプリ（公開）
* Pythonアプリ →（プライベート）→ Lambda（非公開）
* Lambda →（プライベート）→ S3 を読んでユーザ専用ファイルを返す

コードは要らないとのことなので、全体アーキテクチャ、押さえるべき概念、落とし穴をぎゅっとまとめるね。

# 全体アーキテクチャ（推奨構成）

1. ネットワークと入口

* **VPC**：2～3AZ。
* **ALB（Application Load Balancer）**：パブリックサブネットに配置、**ACM**の証明書でTLS終端。
* **EC2 Auto Scaling Group + Launch Template**：プライベートサブネットに配置（インバウンドはALBのみ）。
* **VPCエンドポイント**

  * **Interface**：`com.amazonaws.<region>.lambda`（EC2→Lambdaの**私設経路**確保）
  * **Gateway**：`com.amazonaws.<region>.s3`（Lambda/EC2→S3を**NAT無し**で）
  * これでEC2側は**インターネット不要**でAWSサービスへ到達可能（OS更新等が必要なら別途NATを検討）。

2. アプリ間の呼び出し

* **EC2→Lambda**は**AWS SDKの`Invoke`**を使うのが最短・堅牢。

  * 「プライベートエンドポイント経由」にこだわるなら上記の**Lambda用Interface VPCエンドポイント**を使う。
  * API Gateway Private統合でLambdaを隠す手もあるが、**EC2から直接Invoke**の方が単純で低オーバーヘッド。

3. 認証・認可の流れ（ゼロトラスト的に）

* **エンドユーザ認証**はEC2側（例：Cognito/IdP）で実施。
* EC2は**検証済みのユーザID（subject）**をLambdaに渡す（JWTのクレームを正規化したもの、もしくは内部用Claims）。
* **Lambdaは「ユーザID→S3キー範囲」の対応**を自身で検証する（DynamoDBなどに「ユーザ→許可プレフィックス」やメタ情報を保存）。
* **IAMロール分離**：

  * **EC2ロール**：`lambda:InvokeFunction`（対象関数のARNを**リソース制限**）
  * **Lambdaロール**：`s3:GetObject`（対象バケットの**ユーザ用プレフィックス**のみに制限）、必要なら`kms:Decrypt`
* **Confused Deputy対策**：Lambdaは**EC2から渡されたユーザIDを信用しすぎない**。DynamoDB照合・オブジェクトの**所有者タグ**・キー命名規約など**二重チェック**を行う。

4. S3からの返し方（3パターン）

* **A. 事前署名URL（最もスケーラブル）**

  * Lambdaが**短寿命のS3署名URL**を生成し、EC2へ返す。
  * EC2はクライアントに**リダイレクト**またはURLをJSONで返却。
  * 長所：**データ転送はS3直**、LambdaとEC2のCPU/帯域を消耗しない。
  * 注意：**SSE-KMS**利用時は、**署名者（Lambdaロール）**がKMSに対して`Decrypt`可能で、S3バケットポリシーに`aws:ViaService`条件等を入れておく。
* **B. EC2プロキシ（厳密制御だが重い）**

  * Lambdaは**キーだけ返す**（or 一時クレデンシャル）。EC2がS3から**サーバ側でストリーミング**してクライアントに返す。
  * 長所：**URL露出ゼロ**、細かいレート制御や監査がやりやすい。
  * 短所：EC2の帯域・コスト・**ALB/アプリのタイムアウト**に注意。
* **C. Lambda経由でバイナリ返却（非推奨気味）**

  * 同期Invokeの**レスポンス上限（6MB）**にすぐ当たる。大きいファイルには不向き。

> 実運用は**A**（署名URL）か**B**（EC2プロキシ）。**サイズ・帯域・可観測性要件**で選ぶのがコツ。

# 理解しておくべきAWSの概念

* **PrivateLink / VPCエンドポイント**：パブリックインターネットを経由せず、VPC内からAWSサービスへ**プライベートIPで接続**。Interface型はENIを生やす（従量＋時間課金あり）。S3は**Gateway型**（無料/データ処理は課金なし）。
* **IAMの最小権限**：アクション＋**リソース制限（ARN / prefix）**＋**条件キー（`s3:prefix`, `s3:ExistingObjectTag/*` など）**で絞る。
* **SSE-KMS**：S3の暗号化キーをKMSで管理。**署名URLを使う場合でも、署名者ロールにKMS権限が必要**（S3が背後でKMSを呼ぶため）。
* **分散トレーシング**：**AWS X-Ray**でEC2→Lambda→S3のトレース連携。**相関ID**（`X-Correlation-Id`）をHTTP/Invokeペイロードに埋める。
* **スロットリングと同時実行**：Lambdaの**予約コンカレンシー**、EC2側の**指数バックオフ**、ALB/アプリの**タイムアウト**整合。
* **NAT vs VPCエンドポイント**：OS更新や外部PyPI等が必要なら**NAT**も検討。社内閉域ポリシーなら**CodeArtifact + VPCエンドポイント**でインターネット不要に。
* **S3アクセスポイント / マルチテナント**：ユーザごとの**プレフィックス**、**オブジェクトタグ**、または**S3 Access Point**で論理分離。大規模多テナントなら**Bucket Policy + Condition**で**テナント境界**を機械的に強制。

# セキュリティ設計の勘所

* **クライアントからLambdaを直接呼ばせない**：ネットワーク（PrivateLink）＋IAM（EC2だけが`Invoke`可能）＋アプリ層（Lambda側のユーザ検証）で**三重ガード**。
* **権限の二段ロック**：

  * IAMで**粗い境界**（プレフィックス/ARN制限）
  * アプリで**細かい境界**（ユーザIDとキーの一致、所有者タグ照合）
* **署名URLのTTLを短く**（数分～十数分）＋**一度限りのキー（nonce）**や**DynamoDBに発行履歴**を残すと悪用検知がしやすい。
* **インジェクション/パストラバーサル対策**：S3キーは**正規化**（`user_id/prefix/object.ext`固定フォーマット）、入力値を**URLデコード→妥当性検証**してから使用。
* **KMSキーのスコープ**：キーの**ポリシー**とバケットポリシーの**`kms:ViaService`条件**を揃える。監査ログ（CloudTrail）で追えるように。
* **秘密情報の取り扱い**：EC2/Lambdaの設定は**Parameter Store / Secrets Manager**へ。**ユーザIDマッピング**はDB管理で**アプリ内ハードコード禁止**。

# 可観測性・SRE

* **ログ**：

  * EC2：CloudWatch Agentで**構造化ログ**（JSON）＋メトリクスフィルタ
  * Lambda：構造化ログ＋`REPORT`行から**Duration/MaxMemoryUsed**収集
* **トレース**：X-Rayで**1つのトレースID**をEC2→Lambda→S3の呼び出しに継承
* **メトリクス**：ALB 4xx/5xx、ターゲットの`TargetResponseTime`、Lambdaの`Throttles`/`Errors`/`Duration`、S3の`First Byte Latency`
* **アラート**：SLOに合わせて**エラー率**と**遅延p95**、Lambdaの**スロットリング**を監視

# パフォーマンスとスケール

* **大きなファイルは署名URL**一択に近い（EC2/Lambdaの帯域節約、6MB制限回避）。
* **プロビジョドコンカレンシー**：低レイテンシ要求が強いならLambdaに設定。
* **バックプレッシャ**：Lambdaスロットリング時は**指数バックオフ＋Jitter**。必要なら**SQSバッファ**で非同期化（ただし今回は同期ダウンロードが主眼なので設計に注意）。
* **ALB/アプリのタイムアウト**：EC2プロキシ型では**ALB idle timeout**、アプリWSGI/ASGIの**送信タイムアウト**、**クライアントのダウンロード時間**を揃える。

# デプロイ／運用

* **Immutableデプロイ**：EC2は**AMIベイク**（EC2 Image Builder、Packer）→ASGの**ローリング**。
* **ユーザデータ依存の最小化**：起動スクリプトは短く、設定は**Parameter Store**から注入。
* **環境分離**：`<env>-<bucket>`や`/<env>/<tenant>/`で**明確な名前・プレフィックス規約**。
* **テスト戦略**：

  * ユニット：認可ロジック（ユーザID→キー範囲検証）
  * 連携：**VPCエンドポイント経由でのInvoke**、KMS含む署名URL発行
  * E2E：大容量ファイルの**途中切断**や**再試行**、TTL切れ動作の確認

# 典型的な落とし穴

* **Lambda応答サイズ6MB**を忘れて「ファイルを直接返す」実装にして詰む。
* **SSE-KMS + 署名URL**で**KMS権限不足**（署名者に`kms:Decrypt`が無くて403）。
* **VPCエンドポイント未設定**でEC2がインターネットに出られず**Lambda/S3に届かない**。
* **権限の過剰付与**（`s3:*`/`lambda:*`のバケット丸ごと・関数丸ごと）は後で必ず事故る。
* **相関IDなし**でEC2とLambdaのログが**突き合せ不能**。
* **ALB/アプリのタイムアウト不整合**で大きいファイルの**途中切断**。

# バリエーション（要件次第で検討）

* **API Gateway Private + Lambda**：EC2は**Private API**を叩く。RESTのポリシー制御やWAFを使いたい場合に有利。ただしHopが増えレイテンシと運用が重くなる。
* **S3 Access Point / Object Lambda**：オブジェクトを**加工**して返す要件があるなら検討。
* **S3 + CloudFront + オリジンアクセス**：配布主体が**ダウンロード中心**で、細やかな署名・キャッシュを使いたいならCloudFront署名URL/署名Cookieに寄せる（今回は「クライアントはLambda非公開」が主眼なので脇案）。

---

この構成なら「クライアントにLambdaを露出しない」「VPC内からプライベートに呼ぶ」「S3のユーザ別ファイルを安全に返す」の三条件をきれいに満たせる。次に詰めるなら、（1）**返し方の選択（署名URL vs EC2プロキシ）**と（2）**ユーザID→キー範囲の検証方式**（DynamoDBスキーマとタグ運用）を設計ドキュメントに落としていこう。
