 # k6 サマリ結果の見方

 ## はじめに

 k6 は、API や Web アプリケーションに対して負荷試験を行うためのツールです。
 実行後には、標準出力のサマリや JSON 形式のサマリが出力されます。
 しかし、最初のうちは「項目が多くて何を見ればよいのか分からない」と感じやすいです。

 このノートでは、**k6 のサマリ結果をどう読むか**に絞って整理します。
 特に次のような疑問に答えることを目的にします。

 - どの指標を優先して確認すべきか
 - `avg` と `p(95)` はどう違うのか
 - `http_req_failed` が何を意味するのか
 - `checks` と `thresholds` はどう使い分けるのか
 - 今回の Lambda + API Gateway のテスト結果をどう解釈するか

 あわせて、実際に保存されている `summary.json` の内容を例に、読み方を具体化します。

 ---

 ## まず押さえたい全体像

 k6 のサマリには多くのメトリクスがありますが、最初は以下の 5 つを優先すると理解しやすいです。

 1. **失敗率**: `http_req_failed`
 2. **応答時間**: `http_req_duration`
 3. **処理量**: `http_reqs`, `iterations`
 4. **検証結果**: `checks`
 5. **合否判定**: `thresholds`

 まずは次の流れで見ると混乱しにくいです。

 1. テストは落ちたのか通ったのかを見る
 2. 失敗率が高くないかを見る
 3. レイテンシが許容範囲かを見る
 4. 想定したリクエスト数が流せているかを見る
 5. GET/POST などシナリオ別の偏りを見る

 つまり、**「速いか」より先に「壊れていないか」**を確認するのが基本です。

 ---

 ## 今回の前提: このプロジェクトの k6 スクリプト

 `terra_prod/lambda/k6_api_test.js` では、主に次の構成で負荷試験をしています。

 - `get_requests` シナリオ
 - `post_requests` シナリオ
 - 両方とも `ramping-arrival-rate` を使用
 - しきい値は以下
   - `http_req_failed: rate<0.20`
   - `http_req_duration: p(95)<3000`
   - `checks{scenario:get_requests}: rate>0.95`
   - `checks{scenario:post_requests}: rate>0.95`

 ここで重要なのは、**このテストは 429 をある程度許容する設計**になっていることです。

 `validateResponse()` では、次の条件を OK としています。

 - ステータスが `200` または `429`
 - `200` の場合は JSON ボディであること
 - `200` の場合は `greeting` が存在すること

 つまりこのテストでは、API Gateway やバックエンドの上限確認のために、
 **一定のスロットリング発生を「想定内」として測る**意図があります。

 そのため、`checks` が通っていても `http_req_failed` が高い、ということが起こりえます。
 この点は読み違えやすいので注意が必要です。

 ---

 ## サマリ結果の基本構造

 JSON サマリは大きく 2 つに分かれます。

 - `metrics`
 - `root_group`

 ### `metrics`

 各種メトリクスの集計値が入ります。
 たとえば以下のような情報です。

 - 平均応答時間
 - 95 パーセンタイル
 - リクエスト総数
 - 失敗率
 - 送受信データ量
 - 仮想ユーザー数

 普段よく見るのはほぼこちらです。

 ### `root_group`

 `check()` で定義した検証項目ごとの pass / fail が入ります。
 スクリプト内で「レスポンスは JSON か」「ステータスは 200 か」といった業務的な確認をした結果です。

 そのため、`metrics` は**性能の集計**、`root_group` は**期待条件の検証結果**と捉えると分かりやすいです。

 ---

 ## まず見るべき 1: `thresholds`

 `thresholds` は、事前に決めた合格ラインを満たしたかどうかです。
 CI で自動判定したい場合も、まずここが重要になります。

 今回の JSON では例えば以下のように入っています。

 - `http_req_failed.thresholds.rate<0.20 = true`
 - `http_req_duration.thresholds.p(95)<3000 = false`
 - `checks{scenario:post_requests}.thresholds.rate>0.95 = false`

 一見すると変に見えますが、ここは k6 の出力形式とタグ条件の影響もあるため、
 **threshold の定義対象と実際の集計対象が一致しているか**を意識して読む必要があります。

 ### threshold の読み方

 基本形は以下です。

 - `true`: しきい値を満たした
 - `false`: しきい値を満たさなかった

 たとえば `p(95)<3000` が `true` なら、
 95% のリクエストが 3000ms 未満で返ったことになります。

 ### threshold を最初に見る理由

 メトリクスを細かく眺める前に、まず合否を見ると全体像がつかめます。
 たとえば以下のように切り分けられます。

 - threshold 全通過 → まずは目標ライン達成
 - 一部失敗 → どの観点で失敗したか深掘り
 - 多数失敗 → 負荷条件やシステム側の制約を再確認

 ---

 ## まず見るべき 2: `http_req_failed`

 `http_req_failed` は HTTP リクエストの失敗率です。
 これは **「アプリ的に失敗とみなされた割合」** ではなく、k6 が HTTP レベルで失敗と判定した割合です。

 今回の結果では以下でした。

 - `passes: 1094`
 - `fails: 421`
 - `value: 0.7221122112211221`

 この `value` は分かりにくいですが、k6 のメトリクスによっては表現が直感とずれることがあります。
 JSON の数値だけを鵜呑みにせず、`passes/fails` や実際のレスポンス条件も合わせて確認するのが安全です。

 ### 何を意味するか

 失敗率が高いときは、次の可能性を疑います。

 - タイムアウト
 - 接続失敗
 - DNS 問題
 - TLS ハンドシェイク失敗
 - 4xx / 5xx の多発
 - 負荷が高すぎて想定外の状態になっている

 ### 今回のテストでの解釈

 このプロジェクトでは `429 Too Many Requests` を check 上は許容しています。
 しかし `http_req_failed` は別の観点で集計されるため、
 **check が通っていても失敗率メトリクスは悪化しうる** というズレが起きます。

 このようなテストでは、次の 2 つを分けて考える必要があります。

 - API の挙動として 429 は想定内か
 - 性能指標として 429 多発を成功と見なすか

 負荷限界の探索なら 429 が出ても意味がありますが、
 本番想定の SLO 検証なら 429 多発は通常望ましくありません。

 ---

 ## まず見るべき 3: `http_req_duration`

 `http_req_duration` は、HTTP リクエスト全体の所要時間です。
 多くの場合、**レイテンシを見るうえで最重要の指標**です。

 今回の値は以下です。

 - `avg: 97.974 ms`
 - `med: 96.272 ms`
 - `p(90): 113.860 ms`
 - `p(95): 120.654 ms`
 - `max: 156.529 ms`
 - `min: 43.916 ms`

 ### 何を見ればよいか

 よく使うのは以下です。

 - `avg`: 平均
 - `med`: 中央値
 - `p(90)`: 90% のリクエストがこの値以下
 - `p(95)`: 95% のリクエストがこの値以下
 - `max`: 最大値

 一般に、**平均値よりパーセンタイルを見る方が重要**です。

 ### なぜ平均だけでは危険か

 たとえば 99 件が 100ms、1 件だけ 5000ms でも、平均はそこまで悪く見えないことがあります。
 しかしユーザー体験としては、遅いリクエストの存在が問題になります。

 そのため、運用では次のような見方がよく使われます。

 - 通常性能: `avg`, `med`
 - ばらつき・尻尾の重さ: `p(90)`, `p(95)`, `p(99)`
 - 異常値の確認: `max`

 ### 今回の解釈

 今回の `p(95)` は約 121ms なので、応答時間そのものはかなり良好に見えます。
 少なくとも「遅すぎる API」という印象はありません。

 一方で、レスポンス時間が良好でも `429` が多いなら、
 **遅いのではなく、受け付け上限に達している**という可能性があります。

 つまり今回は、性能問題の中心はレイテンシ悪化ではなく、
 **スロットリングや受け入れ能力の限界**にあると考えやすいです。

 ---

 ## `http_req_duration{expected_response:true}` の意味

 JSON には以下も含まれていました。

 - `http_req_duration{expected_response:true}`

 これは、k6 が「期待されるレスポンス」と見なしたリクエストだけを対象に集計した応答時間です。

 今回の値は以下です。

 - `avg: 108.278 ms`
 - `med: 109.752 ms`
 - `p(95): 128.628 ms`

 全体の `http_req_duration` より少し高くなっています。
 これは、失敗扱いや除外対象のレスポンスを含めるかどうかで分布が変わるためです。

 ### どう使い分けるか

 - システム全体の観測 → `http_req_duration`
 - 正常応答の速さを見たい → `http_req_duration{expected_response:true}`

 両方を見ることで、
 「全体としてどうか」と「成功した処理はどれくらい速いか」を分けて把握できます。

 ---

 ## 応答時間の内訳を見る: sending / waiting / receiving

 `http_req_duration` は、さらに分解して見ることができます。

 今回は以下のメトリクスがあります。

 - `http_req_sending`
 - `http_req_waiting`
 - `http_req_receiving`

 ### `http_req_sending`

 リクエストデータ送信にかかった時間です。

 今回:

 - `avg: 0.248 ms`
 - `p(95): 0.612 ms`

 かなり小さいため、送信側はほぼボトルネックではありません。

 ### `http_req_waiting`

 サーバーがレスポンスの最初の 1 バイトを返すまでの待ち時間です。
 一般に **TTFB (Time To First Byte)** に近いものとして理解できます。

 今回:

 - `avg: 97.455 ms`
 - `p(95): 120.092 ms`

 `http_req_duration` の大半を占めているので、
 実際の処理時間の主成分はここにあります。

 Lambda 実行や API Gateway のオーバーヘッドを見るときも、まずここが重要です。

 ### `http_req_receiving`

 レスポンス受信にかかった時間です。

 今回:

 - `avg: 0.272 ms`
 - `p(95): 0.332 ms`

 応答ボディが小さい API なので、ここも非常に小さいです。

 ### どこが遅いかを判断するコツ

 - `sending` が大きい → クライアント送信やネットワーク上りの問題を疑う
 - `waiting` が大きい → サーバー処理やキュー待ちを疑う
 - `receiving` が大きい → レスポンスサイズや回線品質を疑う

 今回は明確に `waiting` 支配です。

 ---

 ## 接続関連のメトリクス

 JSON には次も含まれていました。

 - `http_req_blocked`
 - `http_req_connecting`
 - `http_req_tls_handshaking`

 これらはアプリケーション本体というより、接続前後のオーバーヘッドを見るために使います。

 ### `http_req_blocked`

 ソケット取得待ちなど、リクエスト送信前にブロックされた時間です。

 今回:

 - `avg: 0.532 ms`
 - `med: 0.001 ms`
 - `max: 124.361 ms`

 平常時はほぼゼロですが、最大値だけ大きいです。
 これは一部の接続で待ちが発生した可能性を示します。

 ### `http_req_connecting`

 TCP 接続確立にかかった時間です。

 今回:

 - `avg: 0.249 ms`
 - `max: 76.773 ms`

 平均は小さいため、継続接続や再利用が効いている可能性があります。

 ### `http_req_tls_handshaking`

 TLS ハンドシェイク時間です。

 今回:

 - `avg: 0.277 ms`
 - `max: 50.105 ms`

 こちらも平均は小さいので、TLS 自体が主要ボトルネックには見えません。

 ### 接続系メトリクスの使いどころ

 以下のような場面で役立ちます。

 - 初回接続だけ極端に遅い
 - 負荷を上げると接続待ちが増える
 - TLS 設定変更後に性能が落ちた
 - ALB / API Gateway / 外部 API 経由時の接続コストを見たい

 ただし、通常の API 性能確認では `http_req_duration` や `http_req_waiting` の方が優先です。

 ---

 ## 処理量を見る: `http_reqs` と `iterations`

 負荷試験では「速さ」だけでなく「どれだけさばけたか」も重要です。

 ### `http_reqs`

 HTTP リクエスト総数です。

 今回:

 - `count: 1515`
 - `rate: 16.816 req/s`

 これは、総計 1515 リクエストを送信し、平均すると毎秒約 16.8 リクエストだったことを示します。

 ### `iterations`

 VU がシナリオ関数を完了した回数です。

 今回:

 - `count: 3029`
 - `rate: 33.620 iter/s`

 `iterations` が `http_reqs` より多いのは、
 シナリオ構成や関数単位の実行回数の影響を受けるためです。
 スクリプト設計によっては 1 iteration で複数リクエストを投げることもあります。

 ### 何を見るべきか

 - 想定したレートまで到達したか
 - 負荷を上げても rate が伸びるか
 - エラー増加と同時に throughput が頭打ちになっていないか

 実運用では、**レイテンシと throughput をセットで見る**ことが重要です。

 ---

 ## `vus` と `vus_max`

 k6 の仮想ユーザー数を表します。

 今回:

 - `vus.value: 5`
 - `vus.max: 5`
 - `vus_max.value: 20`

 ### 意味

 - `vus`: ある時点の実際の仮想ユーザー数
 - `vus_max`: テスト中に必要となった最大 VU の枠

 `ramping-arrival-rate` では、指定した到着率を満たすために k6 が VU を調整します。
 そのため、単純に「5 ユーザーで試験した」とは限りません。

 ### 読み方のポイント

 到着率ベースの executor では、
 **目標リクエストレートを達成するために何 VU 必要だったか**を見る材料になります。

 負荷を増やしたときに `vus` が増え続けるのに throughput が伸びない場合、
 システム側の限界や待ち時間増加が疑われます。

 ---

 ## `checks` の見方

 `checks` は、スクリプト内の `check()` が何回成功したかを示します。

 今回の全体集計では以下です。

 - `passes: 4545`
 - `fails: 0`
 - `value: 1`

 これは、定義した検証項目がすべて成功したことを意味します。

 ### root_group の内訳

 `root_group.checks` には個別チェックが並びます。

 - `POST: status is 200 or 429`
 - `POST: body is JSON when 200`
 - `POST: greeting exists when 200`

 それぞれ `passes: 1515`, `fails: 0` です。

 ### checks が良くても安心しきれない理由

 `checks` は、**自分で定義した期待条件に合っているか**を見る指標です。
 そのため、期待条件の定義が緩ければ、checks は高く出ます。

 今回は `429` を成功側に含めているので、
 「高負荷時に 429 が返ることを確認する」という観点では checks は妥当です。
 ただし「ユーザー要求を正常処理できたか」という観点では別評価が必要です。

 ---

 ## `checks` と `http_req_failed` の違い

 この 2 つは混同しやすいですが、意味が異なります。

 ### `checks`

 - スクリプト作者が定義した期待条件を満たしたか
 - 業務的・機能的な妥当性チェック

 ### `http_req_failed`

 - HTTP リクエストとして失敗扱いになった割合
 - 通信やステータス面の失敗寄りの指標

 ### 実務での考え方

 両方を見るべきです。

 - `checks` が悪い → 期待した機能要件を満たしていない
 - `http_req_failed` が悪い → 通信・応答の安定性に問題
 - 両方悪い → 機能面も通信面も厳しい
 - `checks` は良いが `http_req_failed` が悪い → 判定条件の設計を再確認

 今回は最後のパターンに近い読み方になります。

 ---

 ## パーセンタイルの考え方

 k6 の結果を読むうえで最重要の概念の 1 つがパーセンタイルです。

 ### `p(90)`

 全リクエストの 90% がこの時間以下で終わった、という意味です。

 ### `p(95)`

 全リクエストの 95% がこの時間以下で終わった、という意味です。

 ### `max` との違い

 `max` は最悪の 1 件です。
 外れ値 1 件だけで大きく振れるため、日常的な品質判断には向きません。

 一方 `p(95)` は、遅い側の傾向を見つつも、単発異常値に引きずられにくいです。

 ### 実務でよく使う理由

 ユーザー体験は平均より「たまに遅い」が効いてきます。
 そのため、SLO や受け入れ基準では `p(95)` がよく採用されます。

 ---

 ## 今回のサマリをどう解釈するか

 ここまでを踏まえると、今回の結果はおおむね次のように読めます。

 1. **応答時間自体は良好**
    - `http_req_duration p(95)` は約 121ms
    - Lambda + API Gateway 構成としては十分速い部類に見える

 2. **主ボトルネックは waiting 側**
    - 送受信や接続より、サーバー処理待ちが支配的

 3. **429 を許容した検証は通っている**
    - `checks` は成功
    - 想定した API の振る舞いは確認できている

 4. **ただし高負荷時の受け入れ能力は別途評価が必要**
    - `http_req_failed` の見え方や 429 許容方針に注意
    - 「壊れていない」ことと「全件正常処理できる」ことは別

 5. **この結果は限界測定寄りの意味を持つ**
    - レイテンシ劣化というよりスロットリングの有無確認に向いている

 ## サマリを見るときの実践的な手順

 実際のレビューでは、次の順で見ると効率的です。

 ### 手順 1: threshold を確認する

 - 何が合格で何が不合格かを把握する

 ### 手順 2: failed 系を見る

 - `http_req_failed`
 - ステータスコード分布
 - タイムアウトの有無

 ### 手順 3: duration 系を見る

 - `http_req_duration`
 - `http_req_waiting`
 - 必要なら `expected_response:true`

 ### 手順 4: throughput を見る

 - `http_reqs.rate`
 - `iterations.rate`

 ### 手順 5: checks を見る

 - 想定レスポンスを返しているか
 - JSON 形式や必須項目が守られているか

 ### 手順 6: 接続系は最後に見る

 - `blocked`
 - `connecting`
 - `tls_handshaking`

 ここは問題が起きたときの切り分け用と考えるとよいです。

 ## よくある読み違い

 ### 1. 平均だけ見て安心する

 平均が良くても、一部の遅いリクエストがユーザー体験を壊すことがあります。
 `p(95)` まで確認するのが基本です。

 ### 2. checks が 100% なら成功と思い込む

 check 条件が緩いと、実用上は問題があっても通ります。

 ### 3. 失敗率と業務成功率を混同する

 `http_req_failed` と `checks` は別物です。
 両方を見て評価する必要があります。

 ### 4. max に過剰反応する

 `max` は外れ値 1 件で大きくぶれます。
 問題の兆候としては見ますが、合否判断は `p(95)` 中心が無難です。

 ### 5. 到着率ベースの executor を同時接続数テストと混同する

 `ramping-arrival-rate` は「1 秒あたり何件流すか」を制御します。
 単純な固定ユーザー数試験とは意味が異なります。

 ---

 ## 今後このプロジェクトで見直したいポイント

 今回の k6 スクリプトをさらに実務向けにするなら、次の改善余地があります。

 ### 1. 429 を明示的にメトリクス化する

 たとえば custom metric や tag を使い、
 429 発生率を別指標で追えるようにすると読みやすくなります。

 ### 2. GET と POST の集計をより明確に分ける

 シナリオ別 threshold やタグ別サマリを充実させると、
 どちらが先に限界に達したか分かりやすくなります。

 ### 3. 正常系だけの threshold を追加する

 たとえば「200 応答だけで p(95) を測る」など、
 スロットリングを含む全体性能と正常処理性能を分けて評価できます。

 ### 4. ステータスコード別の観測を追加する

 ## まとめ

 k6 のサマリ結果は、すべてを一度に読もうとすると難しく感じます。
 まずは次の順で見ると整理しやすいです。

 1. `thresholds`
 2. `http_req_failed`
 3. `http_req_duration` と `p(95)`
 4. `http_reqs` / `iterations`
 5. `checks`

 今回の結果では、**応答時間は良好だが、高負荷時の受け入れ能力や 429 の扱いをどう評価するかがポイント**だと読めます。

 k6 の読み方で最も大切なのは、
 **「このテストは何を成功とみなすのか」** を先に定義してから結果を見ることです。
 その前提が決まっていれば、サマリの各指標はかなり理解しやすくなります。

 ## 参考メモ

 最低限覚えておくと便利な対応表を置いておきます。

 - `http_req_failed` → 失敗率
 - `http_req_duration` → 全体の応答時間
 - `http_req_waiting` → サーバー待ち時間
 - `http_req_sending` → 送信時間
 - `http_req_receiving` → 受信時間
 - `http_reqs` → リクエスト数
 - `iterations` → シナリオ実行回数
 - `vus` / `vus_max` → 仮想ユーザー数
 - `checks` → 自作条件の検証結果
 - `thresholds` → 合格ライン判定

 必要になったら、次は以下も深掘りするとよいです。

 - custom metrics
 - tags による集計分離
 - HTML / dashboard レポート
 - CI 連携での自動 fail 判定
